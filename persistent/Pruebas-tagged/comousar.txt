En esta carpeta se hicieron pruebas para etiquetar muestras en GNU Radio que no terminaron funcionando. De cualquier forma, se describe a continuación cómo sería el uso de estos archivos.

La idea es ejecutar con GNU Radio el flowgraph "tagging.grc". En el mismo se tiene una fuente de señal que genera las muestras de un coseno, luego un bloque de Python definido por nosotros, y finalmente un bloque PUB que debería desviar las muestras etiquetadas hacia una conexión socket. El objetivo inicial era poder subscribirse a dicho socket con el script sub_prueba.py, que escribiría las muestras etiquetadas en el archivo taps.txt. 

Se tuvieron varios problemas al intentar hacer funcionar esto. El primero es la falta de documentación útil sobre cómo etiquetar muestras en GNU Radio. El bloque de Python (que debe ser editado desde GNU Radio, más allá de que en esta carpeta se genera una copia llamada "epy_block_0.py") intenta etiquetar las muestras, pero nunca se logró hacerlo (o visualizar que se hizo). No es claro cómo imprimir muestras+etiquetas. Tampoco es claro si el bloque PUB efectivamente pasa las etiquetas al agregar el parámetro "Pass tags". Se cree que sí, pues el script de subscripción genera errores de cantidad de valores recibidos, o de tamaño de buffer (que tampoco pudieron ser corregidos).

No se pudo resolver el etiquetado de las muestras. La idea inicial era poder agregar etiquetas a las muestras en "potumbral", para pasarlas hacia el transmisor etiquetadas. Dado que no se pudo lograr, se utilizó como alternativa la creación de un socket desde potumbral. 

Se intentó entonces pasar desde el transmisor a través de un socket un aviso de que se quería acceder a un slot en ese momento. Como el embeded python block se trancaba con un sub y un pub en el mismo script, se le agregó una entrada al embeded y se le conectó un sub source. El sub del flowgraph se conecta al pub que está dentro del script del transmisor. No se logró un funcionamiento adecuado, lo que se logró enviar desde el embeded python block fue con mucho retardo. Esto generaría muchos problemas pues el transmisor estaba recibiendo información "vieja" y tomaría deciciones sobre slots actuales con los datos de slots viejos.
